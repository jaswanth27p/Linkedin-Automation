# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `bun install` — install deps (postinstall may prompt to trust `agent-browser`/`playwright-core`; run `bun pm trust agent-browser playwright-core` if `bun pm untrusted` lists them, so Playwright's browser binary install runs). Also applies `patches/playwright-core@1.61.1.patch` (see below).
- **Playwright + Bun CDP patch (`patches/playwright-core@1.61.1.patch`, wired via `patchedDependencies`):** Playwright's bundled `ws` (`require("./utilsBundle").ws`) is patched to the standalone `ws` package (`require("ws")`, an explicit dep). Under Bun 1.3.x the bundled `ws` mis-handles the WebSocket upgrade (HTTP 101) and throws `Unexpected server response: 101` instead of firing `open`, so `chromium.connectOverCDP()` hangs and every agent's browser attach dead-ends with a generic "Failed to connect via CDP". Node is unaffected — this is Bun-specific. Do NOT remove `ws` from deps or the patch. `bun patch --commit` fails on Windows with `EPERM ... renaming patch file to patches dir`, so the patch file was authored by hand (via `git diff`) rather than by bun — regenerate the same way if playwright-core is bumped.
- `docker compose up -d` — starts Redis + Postgres, mapped to host ports `6380`/`5433` (not the defaults — deliberately remapped so this project doesn't collide with other local Postgres/Redis instances; `.env`/`.env.example` already point at these ports).
- `bun run db:push` — push Drizzle schema to Postgres (`drizzle-kit push`).
- `bun run dev` / `bun run start` — run the app (`bun --env-file=.env run src/index.ts`).
- `bun run typecheck` — `tsc --noEmit`.
- `bun test` — run the full suite (Bun's native test runner, not vitest/jest).
- `bun test tests/unit/commands/dispatch.test.ts` — run a single test file. `bun test -t "name"` filters by test name.
- Local data files: `cp resume.example.md resume.md` and `cp profile.example.json profile.json`, then fill in real details. Both are gitignored (contain personal data).

## Architecture

Single Bun process — TUI, browser session, command dispatch, and all three agents/queues run in one process. No separate HTTP server.

**Runtime constraints (do not violate):**
- Browser is **always visible** (`headless: false`, hardcoded everywhere). No headless config/toggle anywhere in this codebase.
- No LinkedIn/Gmail credentials are ever read, stored, or requested by the app. Login is always manual, in the visible browser window that Playwright opens at startup — LinkedIn in tab 0, Gmail in tab 1. Gmail login is optional (only needed so the external-apply agent can read OTPs/verification links itself instead of asking the human) and never gates `/verify-login`'s unlock, which still checks LinkedIn only.
- Bun throughout — scripts use `bun run`, not `tsx`/`node` (except the browser subprocess, which Bun spawns as a plain Node child process — see below).
- Postgres only, via `drizzle-orm/node-postgres` (`src/db/`). No SQLite/libsql.
- Test runner is `bun test`, not vitest — vitest's oxc transform rejects the `jsx: "preserve"` + `jsxImportSource: "@opentui/solid"` tsconfig combination this project requires. `@opentui/solid` ships a `bun-test-node` shim specifically for `bun test`.

**Browser bootstrap (`src/browser/session.ts` + `scripts/browser-server.mjs`):** the main process does **not** drive Playwright directly. At startup it spawns `scripts/browser-server.mjs` as a Node child process (`Bun.spawn`), which launches a visible `chromium.launchPersistentContext` and exposes it over a tiny local HTTP API (`/navigate`, `/newtab`, `/check-selector`, `/page-url`, `/cdp-url`, `/state`, `/close`) on a port it reports back via stdout (`READY:<port>`). `session.ts` talks to that subprocess over `fetch()`. `getSharedCdpUrl()` returns the browser's real CDP endpoint (discovered by polling the Chromium profile's `DevToolsActivePort` file) — this is what later agent phases attach to via `@mastra/agent-browser`'s `AgentBrowser({ cdpUrl, scope: 'shared' })`, so every agent shares the one browser you logged into. This subprocess/HTTP-API design is a deliberate deviation from the original plan (which had the main process call `agent-browser`'s `BrowserManager` directly) — see `docs/superpowers/plans/2026-07-14-phase1-foundation.md` Task 5 for the superseded version and `docs/superpowers/specs/2026-07-14-tui-rebuild-design.md` for the still-current data/queue/phase design. `saveState()` in `browser-server.mjs` must `await browser.cookies()` — it's an async Playwright call; calling it without `await` silently serializes the pending Promise (`{}`) instead of the cookie array, which is why login state can silently fail to persist across restarts if this regresses.

`launchPersistentContext` passes `channel: 'chrome'` (uses the system-installed Chrome stable binary, not Playwright's bundled Chromium — needed so Google's sign-in doesn't show "This browser or app may not be secure"), plus `args: ['--disable-blink-features=AutomationControlled']` and `ignoreDefaultArgs: ['--enable-automation']` to strip the `navigator.webdriver` automation fingerprint that same check looks for. `--no-sandbox` was deliberately removed from `args` — it's a Linux-root/Docker concern, not Windows, and its only effect here was an unnecessary "unsupported command-line flag" banner. `openLoginTabs(linkedinUrl, gmailUrl)` opens LinkedIn in the default tab (index 0) via `/navigate` and Gmail in a second tab (index 1) via `/newtab` — tab indices are load-bearing: `verifyLogin()` and the external-apply agent's Gmail-tab-switch both assume this fixed ordering.

**State (`src/state/`):** a Solid store (`app-state.ts`) is the single source of truth — `session` (login status), `activeTab`, per-tab `TabState` (status/step/logs/needsInputQuestion), and `settings`. All mutation goes through its exported setters (`setSessionStatus`, `pushLog`, `setAgentStatus`, `setNeedsInput`, `setSetting`), never by reaching into the store directly — `@opentui/solid`'s reactivity depends on going through the store's own setter (`produce`/path-set), not `.push()` on a store-derived array.

**Commands (`src/commands/`):** a name→handler registry (`registry.ts`) scoped `'global' | TabId`. `dispatch.ts` parses `/name args`, rejects unknown commands, and — critically — rejects any non-global command until `isUnlocked()` (i.e. `/verify-login` has succeeded), directing the user back to `/verify-login`. `global-commands.ts` holds the always-available commands (`/help`, `/tab`, `/set`, `/verify-login`, `/exit`); `search-commands.ts` (`/search-urls`, `/search-describe`, `/search-resume`, `/stop-search`), `easy-commands.ts` (`/process-easy-queue`, `/stop-easy-queue`), and `external-commands.ts` (`/process-external-queue`, `/stop-external-queue`) hold the three tab-scoped command sets. All wired into `registerBuiltinCommands()` (`src/commands/index.ts`) — there's no stub-commands module anymore, every planned command is real.

**Needs-input / prompt channel (`src/state/prompt-channel.ts`):** generic, tab-scoped human-in-the-loop mechanism, used by all three agents. `waitForAnswer(tab, question)` sets the tab's `needsInputQuestion` (sidebar highlights it) and returns a promise that resolves when `answerPrompt(tab, answer)` is called. `InputBar` checks `appState.tabs[activeTab].needsInputQuestion` and switches from command mode to plain-text answer mode whenever it's set, routing the typed reply to `answerPrompt` instead of `dispatchCommand`. Search's `request-human-input` and both apply agents' `ask-human-and-remember` tools all await `waitForAnswer` to pause on a genuinely stuck case (LinkedIn checkpoint, CAPTCHA, unknown form question, email verification) instead of crashing.

**TUI (`src/tui/`):** `@opentui/solid`, styled after opencode's theme (`theme.ts`). `App.tsx` composes `Header` (session status) / `Sidebar` (per-agent status, scrollable, collapses on narrow terminals) / `LogPanel` (active tab's log lines) / `InputBar` (command input, disabled with a placeholder until login is verified). Tab/Shift+Tab cycles the three tabs (`search`/`easy`/`external`); **Ctrl+Q** exits (`/exit`). **Ctrl+C copies the current selection** (App.tsx `copySelection` → `renderer.copyToClipboardOSC52`), and a drag-selection auto-copies on release (`useSelectionHandler`) with a "Copied to clipboard" toast (`components/Toast.tsx`). This requires `createCliRenderer({ exitOnCtrlC: false })` (`index.tsx`) — otherwise opentui's built-in Ctrl+C handler destroys the renderer before `useKeyboard` sees the key. Selection uses opentui's OWN mouse selection (mouse reporting left on, the default), NOT terminal-native selection — the latter can't work in a 30fps-repainting TUI. **Layout must be sized off `useTerminalDimensions()`, not `height="100%"`:** the root box binds `width={dimensions().width} height={dimensions().height}` (like opencode) — `height="100%"` resolved against a root height that was wrong on first paint, so the whole layout was broken (content spilling below the TUI, empty bottom half with a stray scrollbar) until a terminal resize delivered real dimensions. Fixed-height chrome (`Header`, `InputBar`) is wrapped in `flexShrink={0}` boxes; both `scrollbox`es also pass `viewportCulling={false}` and App.tsx nudges `renderer.requestRender()` on log-length change as belt-and-suspenders. `index.tsx` owns the `CliRenderer` lifecycle (`mountTui`/`destroyTui`) — must stay `.tsx` (Bun's transform can't parse JSX in a plain `.ts` file).

**Config & profile:** `linkedin-auto.config.ts` (zod-validated via `src/config/schema.ts`/`loader.ts`) holds search URLs, requirements text, concurrency, model, and `profileFiles` paths — no `headless` field, ever. `src/config/current.ts` holds the loaded `AppConfig` as a module-level singleton (`setCurrentConfig`/`getCurrentConfig`, set once in `src/index.ts`) so commands/agents can read `mustCheckUrls`/`requirements`/`profileFiles` without threading config through every call. Note `model` and `search.irrelevantBailRatio` are also copied into `appState.settings` at startup and are the ones agents should actually read at call time (not the static config) — `/set model <x>` / `/set irrelevantBailRatio <n>` only update the state copy, so live overrides only take effect if code reads from `appState.settings`, not from `getCurrentConfig()`. `resume.md` (free text) + `profile.json` (structured, zod-validated via `src/profile/`) are the user's real data, gitignored, loaded fresh at the start of every run (not just at startup) so edits take effect immediately; `profile.json.answers` is the durable store for learned form-answer Q&A, written by both apply agents' `ask-human-and-remember` tool.

**DB (`src/db/`):** Drizzle schema (`jobs`, `applications`, `search_runs`). `jobs`/`search_runs` are written by the search agent (`src/agents/search-agent.ts`); `applications` is written by both apply agents (`easy-apply-agent.ts`, `external-apply-agent.ts`). Job dedupe across repeated search runs works by primary key: `jobs.id` is the job's numeric LinkedIn id, checked via the `check-already-seen` tool before any relevance judgment is spent on it. Both `processEasyApplyJob`/`processExternalApplyJob` re-check `jobs.status` themselves before doing any work, so a job requeued or double-delivered by BullMQ is a no-op, not a duplicate application.

**Queues (`src/queues/`):** `connection.ts` exposes plain Redis connection options — not an `IORedis` instance, since BullMQ vendors its own `ioredis` and a cross-version instance fails to typecheck. `apply-queues.ts` exposes the producer side (`enqueueApplyJob(applyType, jobId)`, `getApplyQueueCounts(applyType)`); the search agent enqueues relevant jobs to `easy-apply`/`external-apply` as it finds them. `easy-apply-worker.ts` and `external-apply-worker.ts` are the consumers — one BullMQ `Worker` each (`concurrency: 1`), started/stopped by their respective `/process-*-queue`/`/stop-*-queue` commands, wrapping `processEasyApplyJob`/`processExternalApplyJob`. Both compute queue-remaining and applied-today counts per job (the external one also the current job's site domain) and fold them into `setAgentStatus(tab, 'running', step)` — no dedicated sidebar state fields for this.

**Mastra (`src/mastra/index.ts`):** a real `Mastra` instance backed by `@mastra/pg`'s `PostgresStore` (requires an `id` in its config, easy to miss; also exported standalone as `storage` in case something needs the store directly without going through `mastra`). **Search agent (`src/agents/search-agent.ts`):** a singleton `@mastra/agent-browser` `AgentBrowser` (`{ cdpUrl: getSharedCdpUrl(), scope: 'shared', headless: false, excludeTools: ['browser_screenshot'] }`) shared across runs, wrapped by a fresh `@mastra/core` `Agent` per run (cheap — rebuilding it is how each run picks up the latest `resume.md`/`profile.json`/live settings). The agent drives its own browsing via `AgentBrowser`'s tool-calling loop; four custom tools (`createTool` from `@mastra/core/tools`) keep it bounded: `check-already-seen` (dedupe gate), `report-job-verdict` (DB upsert + queue enqueue + scanned/skipped bookkeeping; its `continue` return is a hard mid-page stop only — abort or the per-run job cap, via `computeMidPageContinueDecision()` — NOT the bail ratio, so one skip never cuts a page short), `check-page-bail` (called once per page, after every card in the current traversal list has been reported, to decide via `computeNextPageDecision()` — the bail-ratio math — whether to load more results for this search URL or stop), `request-human-input` (routes through the prompt channel above). Both decision functions are pure and unit-tested — keep them side-effect-free if you touch them. `/search-describe` and `/search-resume` use a second, separate small `Agent` (no browser, just a `return-urls` tool) to turn free text / `resume.md` into search URLs first, then feed them through the same scan loop.

**Search agent browses like a real user, not an API (`buildScanInstructions`):** each given/generated search URL is opened in a fresh browser tab (`browser_tabs` "new") on the real `/jobs/search` UI — left-hand job card list, right-hand detail pane for whichever card is selected, first card auto-selected on load. The agent selects cards one at a time (`browser_click`), reads the already-visible detail pane (no separate `/jobs/view/<id>` page load), extracts the job id from the `currentJobId` URL query param (falls back to the card's `data-occludable-job-id`/`data-job-id` attribute), and calls `check-already-seen`/`report-job-verdict` per card exactly as before. When the loaded cards run out and `report-job-verdict` still says keep going, it clicks the pagination control to load the next page in the same tab; a `continue: false` verdict closes the tab and ends that URL entirely. `browser_click` is throttled by the same `NAVIGATION_TOOLS`/`logSearchStep` rate-limit pause as real navigations — deliberate, it doubles as human-like per-job reading time. This replaced an earlier design that scraped LinkedIn's anonymous `jobs-guest/seeMoreJobPostings` API to list jobs (see `linkedin_guest_jobs_endpoint` in the assistant's memory, now marked superseded) — the user explicitly wants the agent clicking through the normal UI like a person, not hitting an API endpoint.

**Easy-apply and external-apply agents (`src/agents/easy-apply-agent.ts`, `src/agents/external-apply-agent.ts`):** near-identical siblings, each with their own singleton `AgentBrowser` (separate instances, same CDP endpoint), one `Agent` built fresh per job. Field-filling precedence is spelled out in the instructions, not enforced in code: structured `profile.json` fields used directly → `lookup-learned-answer` tool (fuzzy match via `src/profile/answer-matching.ts`'s `findLearnedAnswer`) → LLM inference from resume/profile context → `ask-human-and-remember` tool (prompt channel + persists via `saveLearnedAnswer`, so a question is never asked twice). The external agent additionally has `ask-human-for-verification` — a separate tool, not a flag on the same one, so the model can't forget to opt out of persistence — for CAPTCHAs, SMS/2FA, and any flow it can't resolve itself, which are per-run and deliberately never saved. For email OTPs and confirmation links, the agent is instructed to switch to the Gmail tab itself (`browser_tabs`) and read the code/link directly rather than asking the human — `ask-human-for-verification` is only the fallback if the email hasn't arrived yet or is ambiguous. This only works if Gmail is actually logged in (tab 1, optional — see Runtime constraints); if it isn't, the agent will find an empty/signed-out Gmail tab and should fall back to asking the human. A `report-submission` tool takes `{ success, error? }` from the model and, on success, captures the screenshot itself via `browser.screenshot(...)` called directly (not exposed as a model-facing tool) so the image never enters the (text-only) model's context — writes `applications` + updates `jobs.status`. Neither `processEasyApplyJob` nor `processExternalApplyJob` ever rethrows; if `agent.generate()` throws or finishes without the agent calling `report-submission`, it writes a `failed` row itself, so the worker always moves on. The external agent's instructions additionally tell it to open the apply URL in a **new** browser tab (`browser_tabs` action `"new"`) so the LinkedIn tab it shares the browser with is never navigated away from.

**Gotcha — any `Agent` with `browser` set needs `noOpBrowserContextProcessor` (`src/agents/no-op-browser-context-processor.ts`):** Mastra auto-attaches a `BrowserContextProcessor` to every browser-enabled agent, and it hard-requires Mastra `memory` plus an active `resourceId`/`threadId` on every call — it throws `[Processor:browser-context] computeStateSignal requires Mastra memory with an active resourceId and threadId` otherwise. Confirmed by reading `@mastra/core`'s `getInputProcessors()` source: it skips auto-adding its processor if the agent's own `inputProcessors` already contains one with `id: 'browser-context'`. All three browser-attached agents pass `inputProcessors: [noOpBrowserContextProcessor]` — trying to satisfy the real requirement with a `Memory` instance instead pulls in `@mastra/memory`, which as pinned (`1.22.0` against `@mastra/core@1.48.0`) fails at import with `Export named 'parseDuration' not found` — a real version-drift bug in that combination, not a config issue. Model policy: see `model_choice_policy` in the assistant's memory — default `opencode-go/deepseek-v4-flash`, `opencode-go/mimo-v2.5` only for agents that truly need vision.

**Gotcha — `agent.generate()` defaults `maxSteps` to 5 and fails silently:** all three agents loop over multiple items in a single `generate()` call (search: many job cards per URL; both apply agents: many form fields per job), and Mastra's default cap of 5 tool-call steps total is nowhere near enough — the agent just stops early with no error, which looks exactly like a prompt-following problem but isn't. All three `agent.generate()` calls pass an explicit `maxSteps` sized to the work (search: `Math.max(80, maxJobsPerRun * 10)`; easy-apply: `150`; external-apply: `220`, higher because it adds account creation + a Gmail tab-switch round trip for OTPs on top of the form). These are deliberately high-but-bounded, not unbounded — `maxSteps` is the ONLY circuit breaker against a genuinely stuck agent (e.g. repeatedly retrying the same failed click); neither `easy-apply-worker.ts` nor `external-apply-worker.ts` sets a BullMQ job timeout/`lockDuration` (both just `concurrency: 1`), so an uncapped runaway loop would burn tokens and block every other queued application indefinitely. Don't remove these or bump them to something effectively-infinite thinking they're unnecessary tuning.

## Status

All four phases (TUI shell + login bootstrap, search agent, easy-apply agent, external-apply agent) are implemented. `docs/superpowers/specs/2026-07-14-tui-rebuild-design.md` is the design source of truth and notes where the real implementation deviated from what was originally specced. Nothing has had a real end-to-end run against live LinkedIn/external sites yet — `bun test` covers unit/DB-integration logic only, by design (see the spec's Testing section for why there's no automated e2e against real LinkedIn).
